## Key Method
Adds per-document LoRA test-time training (TTT) at evaluation time. Training is identical to the naive baseline. For each validation document: find document boundaries via BOS tokens, split into overlapping 256-token chunks within 1024-token windows, score each chunk then train rank-8 LoRA adapters on that chunk's loss (no leakage -- only train on already-scored tokens), reset LoRA between documents.

Key ablation insight: most of the improvement comes from document isolation and strided evaluation, NOT the actual LoRA TTT:
- Doc-isolated: -0.0110 bpb
- + Stride (chunk=256): -0.0337 bpb (cumulative)
- + LoRA TTT: -0.0368 bpb (cumulative)
So TTT itself adds only ~0.003 bpb on top of strided eval.

## Components Changed
- Architecture: no -- same baseline model
- Quantization: no -- standard int8+zlib
- Training: no -- identical to baseline
- Evaluation: yes -- per-document LoRA TTT with rank-8 adapters on lm_head, c_q, c_v; Adam lr=0.01, betas=(0.9,0.95); batch_size=64 documents sorted by length; 1 gradient step per chunk
- Compression: no
- Token features: no
- Initialization: no

## Reported Metric
val_bpb: 1.1928 (mean of 4 runs, std=0.0005)
val_loss: 2.0126
Uses ~1/10th of eval budget (~70s on 8xH100)

## Likely Mechanism
Document isolation prevents cross-document context contamination. Strided evaluation gives each token near-maximum context. LoRA TTT provides marginal per-document adaptation. The dominant win is the evaluation strategy, not the actual test-time training.

## Improvement Category
EVAL

## Interactions
- Stacks with: any model improvement (training is unchanged)
- Conflicts with: other eval strategies that already do strided/sliding window (most of the gain is from strided eval, which sliding window also provides)
- The LoRA TTT component specifically could stack ON TOP of sliding window eval for an additional ~0.003 bpb

## Implementation Complexity
medium -- requires document boundary detection, batched LoRA forward/backward, careful no-leakage scoring

## Worth Testing Locally
yes -- the LoRA TTT idea on top of sliding window could provide a small additional gain (~0.003 bpb). The author notes this is "not heavily optimized" and "planting the TTT seed." More TTT steps, higher rank, or better hyperparameters could unlock more.
